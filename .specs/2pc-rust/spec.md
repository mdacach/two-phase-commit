## Context
TLA+ and Alloy specifications for two-phase commit (no-failure model) already exist. This
spec covers a Rust implementation of the same protocol, tested via deterministic simulation.

## Problem
We need an executable implementation that faithfully implements the protocol described in
`alloy/TwoPhaseCommit.als`, with simulation-based testing that checks the same properties
(Agreement, Validity, Termination) the Alloy spec verifies.

## Goals
- [x] Implement the 2PC protocol as pure state-machine logic in Rust.
- [x] Build a deterministic simulation harness that explores protocol interleavings.
- [x] Verify Agreement, Validity, and Termination via property-based testing.

## Non-Goals
- No real networking or async runtime (simulation only).
- No failure modes (crashes, message loss, timeouts) — future work.
- No persistence.

## Technical Approach

### Architecture: Per-Actor State Machines + Message Passing

Each actor (the coordinator and each participant) is an independent state machine that
communicates exclusively through messages. A `StateMachine` trait defines the interface:

```rust
trait StateMachine {
    /// Process an incoming message. Returns outgoing messages.
    fn on_message(&mut self, msg: &Message, at_time: u64) -> Vec<Message>;

    /// Take a self-initiated action if one is available. Returns outgoing messages.
    fn tick(&mut self, _at_time: u64) -> Vec<Message> { vec![] }
}
```

- `on_message` handles message-driven events (receiving `StartTransaction`, votes,
  decisions). The coordinator initiates the protocol by responding to a `StartTransaction`
  external event delivered as a message.
- `tick` handles self-initiated events (deciding, broadcasting decision).
  Default is a no-op — participants are purely reactive and don't override it.
- `at_time` is unused in the no-failure model but is part of the interface so it doesn't
  need to change when failure modes (timeouts, retransmission) are added later.

This trait follows the same shape as `../reliable-channel`'s `StateMachine` trait.

The simulator manages a deterministic event queue (min-heap keyed by delivery time),
following the same architecture as `../reliable-channel`. Events are categorized as:

- **External events** — test-orchestration commands not subject to the protocol:
  - `StartTransaction` — inject a transaction request to the coordinator
  - `Tick { to }` — advance time for one actor
  - `TickAll` — advance time for all actors
- **Internal events** — protocol messages between actors:
  - `Deliver { to, msg }` — deliver a message to its destination

Delivery times for internal events are randomly but reproducibly generated by the
simulator's RNG (`clock + 1 + rng.gen_range(0..jitter_range)`).

### Alloy Event → Actor Method Mapping

| Alloy Event                    | Actor        | Method                             |
|--------------------------------|--------------|------------------------------------|
| `coordinatorSendPrepare`       | Coordinator  | `on_message(StartTransaction)`     |
| `participantVote[n]`           | Participant  | `on_message(Prepare)`              |
| `coordinatorReceiveVote[n]`    | Coordinator  | `on_message(Vote)`                 |
| `coordinatorDecide[d]`         | Coordinator  | `tick()`                           |
| `coordinatorSendDecision`      | Coordinator  | `tick()`                           |
| `participantReceiveDecision[n]`| Participant  | `on_message(Decision)`             |

### Nondeterminism

Alloy's nondeterminism maps to three sources of randomness in the simulation:

1. **Message delivery order** — the event queue orders delivery by `(timestamp,
   sequence_number)`. Delivery timestamps are generated using the simulator's RNG,
   producing reproducible but varied interleavings across seeds.
2. **Participant vote choice** — each participant is seeded with an RNG at construction.
   When receiving Prepare, it draws from its RNG to choose commit or abort. This replaces
   Alloy's disjunction in `participantVote`.
3. **Coordinator abort bias** — the coordinator has a configurable probability
   (`abort_bias`) of aborting even when all votes are commit. The Alloy spec allows the
   coordinator to abort at any point during voting (the guard for
   `coordinatorDecide[Abort]` only requires `phase = Voting`). The implementation exercises
   this by: (a) deciding Abort with probability `abort_bias` when all commit votes are in,
   and (b) spontaneously aborting with some probability on each tick during the Voting
   phase.

### Simulation

The simulator uses a deterministic event queue (min-heap of timestamped events), following
the same architecture as `../reliable-channel`. Each step pops the earliest event, advances
the clock to its timestamp, and processes it. Properties are checked after every step by
inspecting the actors' states via accessor methods.

When delivering a message, the simulator calls `tick()` on the recipient first (implicit
tick), then `on_message()`. Outgoing messages from both calls are enqueued as internal
events with delivery times computed as `clock + 1 + rng.gen_range(0..jitter_range)`.

### Properties

Checked as invariants at every simulation step:

- **Agreement:** For all decided participants, either all committed or all aborted.
- **Validity (Commit):** If coordinator decided Commit, then all nodes voted commit.
- **Validity (Abort):** If coordinator decided Abort, no participant has committed.
- **Termination** *(optional for first pass)*: Under fair scheduling, eventually all
  participants have decided. This may require specific strategies (e.g., appending
  sufficient tick events) and is deferred if non-trivial.

### Testing

Property-based tests with `proptest`, following the event-generation approach from
`../reliable-channel`:

- Generate a sequence of external events (`StartTransaction`, `Tick`, `TickAll`) with
  random time deltas, plus an RNG seed for the simulator.
- Materialize events into monotonically increasing timestamps.
- After the generated events, append a tail of `TickAll` events to ensure the protocol
  can reach termination (analogous to reliable-channel's retransmission tail).
- Run simulation step-by-step, checking Agreement and Validity as invariants after
  every step.
- *(Optional)* Check Termination at the end if fair scheduling was achieved.

## Edge Cases
| Case | Expected Behavior |
|------|-------------------|
| All participants vote Commit | Coordinator decides Commit; all participants commit |
| Any participant votes Abort | Coordinator decides Abort; all participants abort |
| Coordinator aborts despite all commit votes | Valid — Alloy spec permits this |
| Coordinator decides before receiving all votes | Valid (can abort early) |
| Single participant | Protocol completes correctly |
